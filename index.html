Building a personal content aggregator across a homelab, a Mac Studio, and the cloud

01 · Overview

What we're going to build

A personal aggregator that watches the web for me — and tells me what mattered.

Set the destination first. By the end you should know what got built, with what pieces, and where Claude Code fit in. We'll go through motivation, infra, design, ops, and lessons.

What did I need?

Collect

Pull posts from multiple online communities on a schedule.

Store & search

Keep them queryable — by topic, by community, by date.

Summarize & deliver

Daily digest of what mattered, sent to my phone.

Three things I wanted: collect, store-and-search, summarize-and-deliver.

What I already had

  • Homelab — Kubernetes cluster + Argo CD wired up
  • Mac Studio — sitting there waiting to run local LLMs
  • Free GCP VM — small but always-on, public IP — happy to host anything that needs to be reachable from outside
  • Claude Code / Codex — and a willingness to let them drive
I didn't start from zero on infra. Three pieces of hardware and two coding agents — Claude Code and Codex. The GCP VM is more than just a heartbeat-checker: it has a public IP, so anything that needs to be reachable from outside the home network ends up there. The interesting part of the talk is how the agents stitched all of this into something coherent.

What helped most

1 · Plan big, then split into smaller chunks

Slice the plan into agent-shaped tasks — each chunk is either an operation or a piece of code, never both. One session per machine, one task per agent. Keeps purpose unambiguous and dodges content rot.

2 · Ship the bare minimum first, then iterate

Goal v1: "works enough that I could demo it from a United flight." It mostly didn't — so every refinement after was driven by a real failure, not speculation.

3 · Pre-write a concrete CLAUDE.md

The more specific the spec, the closer the first output. Ping-ponging with Claude to refine works, but burns time and tokens. Draft the CLAUDE.md ahead in ChatGPT or Gemini, then hand it over.

4 · Keep agents' lanes from crossing

When multiple agents work in parallel, make sure their work areas don't overlap. Two agents editing the same file or the same cluster object = merge conflicts or silent drift.

Four things to call out. Top row is the conceptual pair — *how to think*: plan big and chunk it down, then ship the bare minimum and let real failures drive iteration. Bottom row is the concrete mechanics — *how to execute*: front-load specificity into CLAUDE.md so the agent doesn't have to guess (using another LLM like ChatGPT or Gemini for the draft is fair game and saves a lot of Claude-Codex ping-pong), and when running multiple agents in parallel, draw clear non-overlapping lanes so they don't step on each other's work.

03 · Infra

The machines I had

Three locations, one VPN, no rented servers.

Before any code, here's the physical reality. This shapes how Claude Code worked across the project.

The hardware

Homelab

Kubernetes cluster. Argo CD already syncing from GitHub. The workhorse.

Mac Studio

Local LLM host. Good for summarization without sending data out.

GCP VM (free tier)

Tiny, always-on, public IP. Perfect for health checks & reverse-proxy hops.

Each box has one job. Homelab = compute, Mac Studio = inference, GCP = the always-reachable foothold.

How they're connected

Home network (MacBook · Homelab · Mac Studio) ⟷ VPN ⟷ External (GCP VM)

The whole thing only works because everything sits on the same Tailscale tailnet. The MacBook reaches the homelab and the Mac Studio as if they were local. Worth calling out: I didn't install Tailscale on any of these boxes by hand — Claude Code did. I had it SSH into each machine in turn (homelab, Mac Studio, GCP VM), run the install, auth the node, and join the tailnet. So even the VPN itself was bootstrapped through the same agent-on-the-box pattern you see in this diagram. Note the agents riding on top of each machine — that's a visual we'll reuse later.

Why the VPN matters

  • One SSH config from the MacBook reaches every node
  • Claude Code can pivot between machines mid-task without leaving the session
  • No public exposure of homelab services — only GCP is exposed

The VPN is what makes "one agent, many machines" actually work.

Without a flat addressable network, multi-machine agent workflows are painful. The VPN was load-bearing.

Two flavors of Airflow DAG run on the homelab. Crawler DAGs pull from each community, land rows in Postgres, and also feed Elasticsearch. The frontend on the GCP VM talks to both: it reads Postgres directly for the post listings, and hits Elasticsearch for the search feature. A separate digest DAG pulls yesterday's window from Postgres, calls the local LLM on the Mac Studio to summarize, and pushes the result to Telegram — all from within the same DAG. The Mac Studio is just an inference endpoint here, not an orchestrator.

The split that unblocked me

"Split this plan by machine, so I can run sessions in parallel."

Homelab session

K8s manifests, Airflow (crawler & digest DAGs), Postgres, ES

Mac Studio session

Local LLM serving — an inference endpoint the DAG can call

GCP session

Web frontend, health checks, reverse proxy, public touchpoints

One session per machine = clean state, no context bleed.

Ask Claude to chunk the design by where work executes. Then run those chunks as separate sessions.

How the split actually happens

One master CLAUDE.md → one prompt → three per-machine CLAUDE.md files.

master CLAUDE.md (excerpt)
# Personal Community Aggregator
## Infrastructure
- MacBook · Homelab (K8s) · Mac Studio · GCP VM
- Tailscale between all four
## Platforms
- Airflow · Postgres · Elasticsearch
- Local LLM (Mac Studio) · Next.js (GCP)
## Where work runs
- Homelab session — K8s, Airflow, PG, ES
- Mac Studio session — LLM serving
- GCP session — frontend, nginx, certs
homelab CLAUDE.md (derived)
# Homelab session
## Scope
K8s cluster only — out of scope: LLM, frontend.
## What lives here
Airflow DAGs · Postgres · Elasticsearch · Argo CD
## External endpoints I call
- mac-studio.tailnet:8080 (NSFW + summary)
## Conventions
- Manifests live in `manifests/` repo, NOT app repos
- Never kubectl apply directly — commit to manifests/
mac-studio CLAUDE.md (derived)
# Mac Studio session
## Scope
Local LLM serving only.
## Endpoints exposed
- POST /v1/classify → { nsfw: bool }
- POST /v1/summary  → { text: string }
## Conventions
- No public exposure — Tailscale only
- Model swap = restart launchd service
- Pre-stage GGUF files in ~/models/

Full examples in examples/ — master + 3 per-machine files.

Concretely: I keep one master CLAUDE.md in the repo root that captures the whole project — goal, infra, schema, platforms, conventions. Then a single prompt — literally "split this by machine and produce one CLAUDE.md per session" — gives me three derived files. Each session only loads its own slice. The homelab session never sees the launchd or nginx details. The Mac Studio session never sees Argo CD. The GCP session never sees the data schema. Three benefits: faster context (no irrelevant tokens), less content rot (changes to the master don't accidentally regenerate stale assumptions in the per-machine files until I ask), and clearer accountability (when something goes wrong, I know exactly which file the agent was reading from).
05 · Operations

Running it day-to-day

Where Claude Code earned its keep.

Design is one-off. Ops is forever. This is where Claude Code shifted from "code generator" to actual operator.

Database work

Schema in CLAUDE.md

What to crawl, what to store, what shape it lands in.

Migrations

Claude generates SQL, I review, it runs against Postgres.

Pruning & normalization

Periodic clean-up queries, dedupe runs, column-type fixes.

[Point at the snippet.] This is the entire data-model section in CLAUDE.md. Notice what's *not* there — I didn't write a single line of SQL. I just listed the fields I knew I needed: title, URL, community, a cross-community global_id, fetched timestamp, vote counts, and a couple of null-until-filled-later columns. Claude picked the column types, picked sensible indexes, and dropped a migration into db/migrations/ on its own. The point of putting this prompt in CLAUDE.md isn't to hand-craft the schema — it's to give every future session the same starting context, so the crawler code and the query code never drift apart. When I need a new field later, I add a line to this list and ask for another migration.

Deployments — the happy path

flowchart LR Dev["MacBook
(Claude/Codex)"] -->|"git push"| App["App repo"] App -->|"Actions"| Build["Docker build"] Build -->|"push image"| Reg["Image registry"] Build -->|"bump image tag
(commit)"| Man["K8s manifest repo"] Man -->|"watch"| Argo["Argo CD"] Argo -->|"sync"| K8s["Kubernetes (homelab)"]

App repo build → push image and commit new tag into the manifest repo → Argo CD watches the manifest repo, syncs the cluster.

GitOps loop. The app repo and the manifest repo are separate. When Actions finishes the build, it does two things: pushes the image to the registry, and commits the new image tag back into the manifest repo. Argo CD is *only* watching the manifest repo — it doesn't poll the registry. So the cluster's desired state always lives in Git, which means re-syncs, rollbacks, and audits are just git operations. For application code, this whole loop is hands-off after the initial push.

The other path — SSH

Not everything fits Git-driven deploys.

  • Resource bumps on a stuck pod
  • One-off DB hotfix or index rebuild
  • Local LLM model swap on the Mac Studio
  • GCP VM tweaks (nginx, certs)

Claude Code with SSH access = a real on-call hands-on operator.

Sometimes you just need to SSH and fix it. Claude Code is great at this because the VPN puts everything one hop away.
06 · Multi-Agent

Two agents, different jobs

Claude Code and Codex, side by side.

The part of the talk I find most fun. The division of labor between two coding agents.

The division of labor

Claude Code

Infrastructure & operations.

SSH into machines. Edit manifests. Run kubectl. Patch DAGs in place. Debug pods.

Codex

Application code.

Airflow DAG implementations. Frontend app. Pure code repos. PRs into GitHub.

Early on, Codex had to wait — Claude needed to set the infra up first.

Claude Code is the one with hands on the machines. Codex is the one with hands on the codebase. No overlap.

What it looks like in motion

Walk through the steps. (1) Claude Code SSHs from the MacBook into the homelab or Mac Studio for ops. (2) In parallel, Codex pushes to the app repo. (3) Actions builds the image and pushes it to the registry. (4) The same Action commits a tag bump into the manifest repo — that's the GitOps trigger. (5) Argo CD is only watching the manifest repo; it sees the new commit. (6) Cluster reconciles, pod rolls. Cluster state always lives in Git.

Why the split worked

Upsides

  • No collision — different layers, never the same file
  • Right tool for the layer — Claude's terminal, Codex's repo
  • Easier review — Codex diffs in GitHub, Claude changes in the cluster
  • Recoverable — Argo CD is the truth; re-sync to recover
  • Token budget spread — two quotas, half the rate-limit pressure
Five reasons the lane split paid off. Walk down the list, but spend the most time on the last one because it's the practical motivation people don't expect. Rate limits: everything through one agent and you hit the daily or weekly cap mid-task — split across two and you get roughly twice the runway. The others are the obvious wins: different layers means no collision, each agent runs in its native tool, the diffs end up where review is easy, and Argo CD gives you a clean "source of truth" to re-sync against when something drifts.

…and where it bites

Downsides

  • Overlap = conflict — if both touch the same area, you get merge or state conflicts
  • No shared memory — each agent remembers only its own work; the other can be blind to it
  • ↳ Mitigation: have the agent re-read the current state from the code repo or the live cluster before acting

Two specialized agents beat one generalist — as long as you keep their lanes from crossing.

Now the downsides, because the split isn't free. First — overlap. If you accidentally let both agents touch the same file or the same cluster object, you get conflicts: either a git merge conflict, or worse, drift between what each agent *thinks* the world looks like. Second — they don't share memory. Each agent's notes and memory live in its own session. So if Claude Code just patched a deployment, Codex has no idea unless it goes and looks.

The mitigation is the same in both cases: don't trust either agent's mental model. Have it re-read the source of truth — the code repo, the live cluster, the actual state — at the start of any non-trivial change.

Then land the callout: two specialized agents beat one generalist — *as long as you keep their lanes from crossing*. The "as long as" matters; without that discipline, the split actively hurts.
07 · Hiccups & Lessons

It did not work the first time

A short tour of things that went wrong.

Honest section. This isn't a "I prompted once and shipped" story.

What broke early on

K8s under-provisioned

Pods up, but resources too tight. OOMs and throttling.

Silent crawls

Logs missing. Failed collections looked like empty days.

Elasticsearch misses

Term extraction option wasn't set — relevant docs were invisible.

Airflow DAG fragility

No parallelism. One failure cascaded into the next day's run.

These were invisible until I started looking. Claude Code helped triage each one — but the human still has to look.

What worked surprisingly well

Generic crawlers

I gave only the URLs. Claude figured out the list parsing for each community on its own.

Schema-driven prompts

Once the DB schema lived in CLAUDE.md, crawler and query code stopped drifting.

Recovers broken systems

After a host reboot it couldn't explain why the box went down — but it spotted that a Postgres data disk failed to mount, edited /etc/fstab, remounted, restarted Postgres, and brought the service back.

Handles gnarly K8s commands & manifests

Runs the long kubectl incantations directly. Comfortable with the awkward manifest bits too — hostPath mounts, GPU mounts, secret management.

Reads error logs & fixes itself

On a failure, it tails the logs, isolates the offending stack frame, and patches the code directly — no hand-holding.

"Just give it the URL" worked far more often than I expected.

I expected to write per-community parsers. I mostly didn't have to. Beyond that, three more things surprised me on the ops side. First, debugging from logs — when something fails, Claude reads the error output, walks the trace, and patches the code without me having to translate the failure into a prompt. Second, recovery after a real outage. One of the homelab nodes rebooted on its own. Claude couldn't tell me why the host went down — that question is out of its reach — but it noticed the Postgres data disk failed to remount, fixed /etc/fstab, brought the mount back, restarted Postgres. End-to-end recovery without me touching the keyboard. Third, Kubernetes. The long kubectl commands and the fiddly manifest fields — hostPath, GPU resources, secrets — are the parts I always have to look up. Claude just does them.

Refinements that mattered

  • Timestamp normalization — store as UTC, render in local TZ
  • HTML selector tuning — extract body, not chrome/ads
  • DAG isolation — one community fails ≠ all fail
  • Resource right-sizing — observe, then bump
  • UI polish — search bar, dark mode, community/tag filters

Each one was a 30-minute conversation with Claude — not a 3-day project.

Notice the smell, describe it, let Claude propose, review, ship. The UI bucket grew over time — once the backend was steady, I started layering on quality-of-life features: a search bar wired up to Elasticsearch, dark mode, and filters by community or tag. Each was a single conversation, not a project.

The result that lands on my phone

From scattered communities to one morning digest.

Local LLM — two jobs

Same Mac Studio, two pipelines: an NSFW classifier on incoming posts, and a two-pass summarizer for the daily digest.

Once the local LLM was sitting on the Mac Studio, it was easy to give it more than one job. Job one — classification. For every new post, the model looks at the title and the body and decides whether it's NSFW. The flag goes back into Postgres alongside the post. Job two — summarization. Same model, two passes: per-community first, then a meta-summary across all of them. The NSFW filter feeds the top-10 selection, so the digest never includes anything I'd rather not have on my phone over breakfast. The lesson here is the local LLM stopped being a "summarizer" — it became a generic inference endpoint I could keep adding tasks to.

What lands on my phone

  • One Telegram message per morning
  • Top topics across my chosen communities
  • Links back to the originals if I want to dive deeper

No catch-up scrolling. No FOMO. Just what mattered.

[Point at the screenshot.] This is an actual morning digest. The header is the date. Then the day's headlines as bullets — usually 5 to 8 across all communities. Below that, per-community highlights. The whole thing fits on one screen, no scrolling needed. If a topic is interesting, I tap through to the original post. The point isn't that the digest is comprehensive — it's that it's enough to decide whether to dig in.

What changed for me

Time

No more 20-min site hopping at lunch.

Coverage

I see more communities now than I used to visit manually.

Signal

Pre-ranked by community engagement, not algorithmic feed.

Discovery via digest

I don't dig through whole threads anymore. The Telegram digest surfaces the popular posts, and I only follow the ones that look worth my time.

Wider net, less time. Reading what humans found interesting, not what a recommendation engine wants me to see. And the discovery model flipped — I used to hunt through every community to find anything interesting. Now I read the digest first, and only pull up the original post if the summary makes it worth following.
09 · Applications

What this unlocks next

The collection pipeline is the foundation.
Everything else is a remix.

Once data flows reliably, every new idea is a small downstream module.

Takeaways

  • Plan first, code second. CLAUDE.md is your contract with the agent.
  • Split tasks into smaller chunks. One session per box keeps state clean.
  • Keep agents' lanes from crossing. If multiple agents work in parallel, separate their areas cleanly — overlap creates conflicts and silent drift.
  • Engineering knowledge still matters. Our background is the edge non-engineers don't have — agents amplify it, they don't replace it.
  • Data is the moat. Once collection runs, every new feature is days, not months.
Five things. Plan with CLAUDE.md as the contract; split tasks small so each session has clean state; if you run multiple agents in parallel, draw clear non-overlapping lanes so they don't step on each other; lean on your engineering background — agents don't make non-engineers into engineers, they make engineers faster; and remember that the data layer is what keeps paying off long after the first ship.
